    4、CyclicBarrier源码解析

    一 CyclicBarrier源码解析

    1.1 CyclicBarrier简介

    CountDownLatch我们已经了解其原理及应用了，JDK中还提供了其他的并发工具，而CyclicBarrier就是其中之一。
    CyclicBarrier，循环屏障，可以让一组线程达到一个屏障时被阻塞，当最后一个线程达到屏障时，才能继续执行。运行完成之后，屏障又会自动重置，所以被屏障拦截的线程又可以继续运行。所以CyclicBarrier是可以重用的。

    1.2 示例

    我们先看一个示例：有三个用户从各自家中去餐厅，由于路程不同，到达餐厅的时间也不同，当所有人都达到餐厅之后开始就餐。吃完饭之后由各自驾车去同一个KTV唱歌，当所有人都到达KTV之后开始唱歌。
    代码如下：

    public class CyclicBarrierTest {
    /** 执行线程池, 指定线程创建工厂 */
    private static final ExecutorService executor = Executors.newFixedThreadPool(10, new ThreadFactory() {
        /** 计数器 */
        AtomicInteger counter = new AtomicInteger();
        @Override
        public Thread newThread(Runnable r) {
            final Thread thread = new Thread(r);
            thread.setName("用户" + counter.getAndIncrement());
            return thread;
        }
    });

    public static void main(String[] args) throws InterruptedException {
        int threadCount = 3;//运行的线程数
        CountDownLatch latch = new CountDownLatch(2);
        //创建CyclicBarrier, 并指定运行的任务
        CyclicBarrier cb = new CyclicBarrier(threadCount, new Runnable() {
            @Override
            public void run() {
                latch.countDown();
            }
        });
        final TaskThread taskThread = new TaskThread(cb);//业务线程
        for (int i = 0; i < threadCount; i++) {
            executor.execute(taskThread);//执行
        }

        latch.await();//等待
        executor.shutdown();
    }

    private static class TaskThread implements Runnable {
        /** 随机数 */
        private static final Random random = new Random(10);

        /** 计数器 */
        private static final AtomicInteger counter = new AtomicInteger(1);

        /** CyclicBarrier */
        private final CyclicBarrier cb;

        public TaskThread(CyclicBarrier cb) {
            this.cb = cb;
        }

        @Override
        public void run() {
            try {
                sleep(random.nextInt(5) + counter.getAndIncrement(), TimeUnit.SECONDS);//此处模拟睡眠, 模拟人从家里到餐厅花费的时间
                System.out.println(Thread.currentThread().getName() + "到达餐厅。当前已有: " + (cb.getNumberWaiting() + 1) + "个人到达餐厅.");
                //等待所有人都到齐了就开始吃饭 Start
                cb.await();
                System.out.println("所有人都达到餐厅了, " + Thread.currentThread().getName() + "开始吃饭了");
                //等待所有人都到齐了就开始吃饭 End

                //分隔符 Start

                //分隔符 End

                //吃完饭各自驾车去KTV唱歌(CyclicBarrier的重用) Start
                sleep(random.nextInt(10) + counter.getAndDecrement(), TimeUnit.SECONDS);//此处模拟睡眠, 模拟人去KTV的时间
                System.out.println(Thread.currentThread().getName() + "到达KTV, 当前已有: " + (cb.getNumberWaiting() + 1) + "个人到达KTV.");
                cb.await();
                System.out.println("所有人都到KTV了, " + Thread.currentThread().getName() + "开始唱歌...");
                //吃完饭各自驾车去KTV唱歌(CyclicBarrier的重用) End
            } catch (Exception e) {
                e.printStackTrace();
            }
        }
    }

    public static void sleep(int timeout, TimeUnit timeUnit) {
        try {
            timeUnit.sleep(timeout);
        } catch (InterruptedException e) {
            e.printStackTrace();
        }
    }
}

    从运行的效果来看，CyclicBarrier是可以重用的，并且是所有线程都达到现场之后会自动运行任务并继续任务。下面我们来详细介绍其原理。

    1.3 构造方法

    CyclicBarrier没有继承任务类和接口：

    public class CyclicBarrier {}

    那么我们先看一下CyclicBarrier的一些全局变量：

    /** 互斥锁 */
private final ReentrantLock lock = new ReentrantLock();
/** 条件队列 */
private final Condition trip = lock.newCondition();
/** 参与者的数量, 通常来说是线程数量 */
private final int parties;
/** 当任务中所有的线程运行完成之后这个线程就触发了 */
private final Runnable barrierCommand;
/** 生成器, 表示当前是否broken */
private Generation generation = new Generation();

/** 等待的参与者数量, 在每次创建后会从{@link #parties}数量减到0. */
private int count;

    从全局变量来看，CyclicBarrier并没有直接使用AQS，而是使用ReentrantLock和Condition完成主要的锁任务，而ReentrantLock与Condition在上文我们也已经介绍过了（2、ReentrantLock源码解析），这里不做重复介绍。并且使用变量parties表示参与者的数量，这个变量是final的，不允许更改。generation是一个可以表示线程持有的生成器，其代码如下：

    /**
 * 屏障的每次使用都有一个Generation实例。无论何时屏障发生了或者reset了, Generation的实例都会改变。
 */
private static class Generation {
    boolean broken = false;//是否broken
}

    CyclicBarrier有两个构造函数：

    /**
 * 构造器
 * @param parties 参与者数量, 通常来说是线程数量。不能小于等于0
 * @param barrierAction 线程, 当任务中所有的线程运行完成之后这个线程就触发了
 */
public CyclicBarrier(int parties, Runnable barrierAction) {
    if (parties <= 0) throw new IllegalArgumentException();//小于等于0的话抛出异常
    this.parties = parties;
    this.count = parties;
    this.barrierCommand = barrierAction;
}

/**
 * 构造器
 * @param parties 参与者数量, 通常来说是线程数量。不能小于等于0
 */
public CyclicBarrier(int parties) {
    this(parties, null);
}

    由构造函数可以看出，初始状态的parties与count是同一个值，即表示线程的数量。而barrierCommand是可以为空的，从后面的代码我们知道，如果barrierCommand为空则不作事件操作，而是直接执行用户的线程。了解了CyclicBarrier的构造过程之后，我们来看一下它的await方法。

    1.4 await等待

    await方法有两种形式：

    • int await()
    • await(long timeout, TimeUnit unit)
    /**
 * 阻塞等待
 * @return int
 * @throws InterruptedException 异常
 * @throws BrokenBarrierException 异常
 */
public int await() throws InterruptedException, BrokenBarrierException {
    try {
        return dowait(false, 0L);
    } catch (TimeoutException toe) {
        throw new Error(toe); // cannot happen
    }
}

/**
 * 等待, 直到超时
 * @param timeout 超时时间
 * @param unit 时间单位
 * @return int
 * @throws InterruptedException 异常
 * @throws BrokenBarrierException 异常
 * @throws TimeoutException 异常
 */
public int await(long timeout, TimeUnit unit) throws InterruptedException, BrokenBarrierException, TimeoutException {
    return dowait(true, unit.toNanos(timeout));
}

    从源码来看不管哪种形式都会委托给dowait(boolean timed, long nacos)方法：

    /**
 * 等到
 * @param timed 是否是超时的等待
 * @param nanos 如果timed是true的话, 表示等待的时长
 * @return int
 * @throws InterruptedException 异常
 * @throws BrokenBarrierException 异常
 * @throws TimeoutException 异常
 */
private int dowait(boolean timed, long nanos) throws InterruptedException, BrokenBarrierException, TimeoutException {
    final ReentrantLock lock = this.lock;
    lock.lock();//加锁
    try {
        final Generation g = generation;

        //如果已经终止, 则抛出异常 Start
        if (g.broken)
            throw new BrokenBarrierException();
        //如果已经终止, 则抛出异常 End

        //如果线程已经终止, 则调用breakBarrier(改变Generation实例)后抛出异常 Start
        if (Thread.interrupted()) {
            breakBarrier();
            throw new InterruptedException();
        }
        //如果线程已经终止, 则调用breakBarrier(改变Generation实例)后抛出异常 End

        //触发回调的屏障线程的run方法 Start
        int index = --count;//index, 所有的线程是否已经完成
        if (index == 0) {//当index为0的时候, 所有的线程已经运行完成, name就需要运行回调的屏障线程了
            boolean ranAction = false;//是否运行成功
            try {
                //运行屏障线程的run方法 Start
                final Runnable command = barrierCommand;
                if (command != null)
                    command.run();
                //运行屏障线程的run方法 End
                ranAction = true;//没有异常证明运行完成
                nextGeneration();//唤醒所有线程并重置count、Generation实例
                return 0;
            } finally {
                if (!ranAction)
                    breakBarrier();
            }
        }
        //触发回调的屏障线程的run方法 End

        /*
         * 如果所有的线程没有运行完成, 那么就需要进一步处理(等待、超时等处理)
         */
        for (; ; ) {//死循环, 进一步处理
            try {
                if (!timed)//没有超时时间的设置
                    trip.await();//阻塞等待(这里会释放队列中所有的锁, 即会将state重置为0, 此时其他线程通过lock.lock()是成功的)
                else if (nanos > 0L)//存在超时时间的设置
                    nanos = trip.awaitNanos(nanos);//阻塞等待指定的时间
            } catch (InterruptedException ie) {//出现异常
                if (g == generation && !g.broken) {
                    breakBarrier();//改变generation实例的状态并唤醒所有的线程
                    throw ie;
                } else {
                    Thread.currentThread().interrupt();//中断线程
                }
            }

            //如果有线程改变了Generation实例的状态了则抛出异常 Start
            if (g.broken)
                throw new BrokenBarrierException();
            //如果有线程改变了Generation实例的状态了则抛出异常 End

            //如果此时有其他线程改变了Generation实例了, 说明不再需要操作了, 直接return操作 Start
            if (g != generation)
                return index;
            //如果此时有其他线程改变了Generation实例了, 说明不再需要操作了, 直接return操作 End

            //超时 Start
            if (timed && nanos <= 0L) {
                breakBarrier();
                throw new TimeoutException();
            }
            //超时 End
        }
    } finally {
        lock.unlock();//释放锁
    }
}

    整段代码其实可以分为三段来解析：

    1. ReentrantLock的加锁、解锁
    2. 是否需要唤醒并触发屏障命令
    3. 阻塞线程

    1.4.1 ReentrantLock解锁、解锁

    ReentrantLock加锁、解锁的原理我们已经在上文 2、ReentrantLock源码解析 中有详细介绍，这里不做重复介绍了。

    1.4.2 唤醒并触发屏障事件

    代码片段如下：

    //触发回调的屏障线程的run方法 Start
int index = --count;//index, 所有的线程是否已经完成
if (index == 0) {//当index为0的时候, 所有的线程已经运行完成, name就需要运行回调的屏障线程了
    boolean ranAction = false;//是否运行成功
    try {
        //运行屏障线程的run方法 Start
        final Runnable command = barrierCommand;
        if (command != null)
            command.run();
        //运行屏障线程的run方法 End
        ranAction = true;//没有异常证明运行完成
        nextGeneration();//唤醒所有线程并重置count、Generation实例
        return 0;
    } finally {
        if (!ranAction)
            breakBarrier();
    }
}
//触发回调的屏障线程的run方法 End

    由count控制线程剩余的数量，如果count为0时，表示所有的线程都已经达到了指定的状态，此时会看barrierCommand是否不为空，不我空则执行其run()方法，并调用nextGeneration()方法后return。如果没有出现异常，则ranAction为true，就不会执行finally{}代码块中的if内容, 即不会调用breakBarrier()方法。
    nextGeneration()与breakBarrier() 总体相似，都是唤醒线程并重置count，代码如下：

    /**
 * 更新Generation实例, 这里主要是可以重入的机制, 多次调用
 */
private void nextGeneration() {
    trip.signalAll();//唤醒所有的线程
    count = parties;//重置count值
    generation = new Generation();//重新生成Generation实例
}

/**
 * 将Generation.broken的状态置为true, 并且唤醒所有的线程
 */
private void breakBarrier() {
    generation.broken = true;
    count = parties;
    trip.signalAll();
}
    • nextGeneration()方法是将generation重置，从上文我们的得知，如果直接new操作的话，generation.broken是为false的。此时可以正常退出方法
    • breakBarrier()方法是将generation.broken置为true, 因为只有运行barrierCommand.run()出现异常了，才会调用该方法，那么在dowait(boolean timed, long nanos)方法的后续中会抛出BrokenBarrierException()异常

    这两个方法都会调用trip.signalAll() 方法，即唤醒所有线程。从 1、AbstractQueuedSynchronizer原理分析-3.3 signal唤醒 中我们得知，其实就是将条件等待队列的节点全部传输到AQS的阻塞队列中，最后由阻塞队列控制其唤醒操作，这里实际上也就是ReentrantLock的unlock()的解锁操作。此时CyclicBarrier的流程其实已经return了。
    但是如果count并不是0的话，证明还有线程未达到屏障状态，那么当前线程就需要等待了，也就是接下来我们说的阻塞。

    1.4.3 阻塞

    阻塞其实调用的是Condition.await() 方法（Condition.await(long nacos)与其类似），而await()方法其实就是AQS的ConditionObject.await()方法，具体在 1、AbstractQueuedSynchronizer原理分析-3.2 await阻塞等待 可以看详解。此时线程进入阻塞状态。
    这里需要注意的是，在Condition.await()方法中会调用fullyRelease(Node)方法，会释放AQS的state值，即将ReentrantLock的锁资源置为了0，此时如果下个线程进来的话，依然是可以ReentrantLock.lock()成功的。
